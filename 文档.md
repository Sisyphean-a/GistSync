## 项目需求

**项目名称**：GistSync (配置文件云端同步工具)

**核心功能**：基于跨平台 GUI 界面，通过 GitHub Gist 备份与同步开发者的本地配置文件（如 ssh 密钥、环境配置等）。

**关键技术点**：

1. 本地文件系统操作与跨操作系统（用户名）路径动态映射。
2. 数据安全性：上传至 Gist 前，必须通过用户设置的本地主密码（Master Password）进行 AES-256 强加密。
3. 架构：Wails (Go 后端 + Web 前端)，跨平台单文件打包。

- **🛠️ 推荐工具/轮子**：`wails init` 命令行工具。
- **🚫 边界限制 (防超纲)**：禁止修改任何自动生成的业务代码，只要能把默认的 Hello World 跑起来即可。
- **✅ 考官验收标准 (PM Checklist)**：

### 🟢 Quest 1.2：绘制配置页面 UI 骨架

- **🎯 关卡目标**：在前端绘制出“设置页面”，包含 GitHub Token 输入框、主密码（Master Password）输入框和保存按钮。

## 阶段 2：安全核心与路径引擎 (Go Backend Core)

*本阶段纯后端，无需界面的配合。目标是让学生利用 Go 语言实现“强加密”以及解决你提到的“不同电脑用户名路径不同”的问题。*

### 🟢 Quest 2.1：AES-256 加解密纯函数

- **🎯 关卡目标**：在 Go 层面编写两个互相可逆的纯函数：输入明文+密码输出密文，输入密文+密码输出明文。
- **🛠️ 推荐工具/轮子**：Go 标准库 `crypto/aes` 和 `crypto/cipher`。
- **🚫 边界限制 (防超纲)**：本关严禁读取任何真实硬盘文件，纯粹只做内存里的字符串转换测试。
- **✅ 考官验收标准 (PM Checklist)**：
  - [ ] 测试动作 1：在终端运行该模块的单元测试代码 `go test -v`。
  - [ ] 预期结果 1：控制台打印出一串乱码（密文）。
  - [ ] 测试动作 2：观察下一步的控制台输出。
  - [ ] 预期结果 2：密文被成功还原为原始的测试字符串，测试绿灯通过 (PASS)。

### 🟢 Quest 2.2：操作系统家目录动态映射器

- **🎯 关卡目标**：编写 Go 函数，将带有 `{{HOME}}` 标识的相对路径转化为当前电脑的绝对路径。
- **🛠️ 推荐工具/轮子**：Go 标准库 `os.UserHomeDir()` 和 `path/filepath`。
- **🚫 边界限制 (防超纲)**：不要处理 Gist 上的网络路径，只处理本地字符串替换。
- **✅ 考官验收标准 (PM Checklist)**：
  - [ ] 测试动作 1：给这个 Go 函数传入测试参数 `{{HOME}}/.ssh/id_rsa` 并运行。
  - [ ] 预期结果 1：(如你在 Windows) 控制台正确打印出 `C:\Users\你的用户名\.ssh\id_rsa`；(如你在 Mac) 打印出 `/Users/你的用户名/.ssh/id_rsa`。传入普通绝对路径时，原样输出不变。

## 阶段 3：GitHub Gist 的云端挂载 (API Integration)

*本阶段需要学生通过查阅 GitHub 官方 API 文档，用 Go 语言把 Gist 当作云盘来使用。*

### 🟢 Quest 3.1：通过 Token 获取/创建云端大纲 (Manifest)

- **🎯 关卡目标**：使用传入的 Token，在 GitHub Gist 寻找或新建一个名为 `sync_manifest.json` 的文件，这是后续存放所有同步文件的“目录大纲”。
- **🛠️ 推荐工具/轮子**：Go 标准库 `net/http` 或第三方库 `go-github`。
- **🚫 边界限制 (防超纲)**：禁止实现具体配置文件的上传，本关只关心这一个 `manifest` 索引文件。
- **✅ 考官验收标准 (PM Checklist)**：
  - [ ] 测试动作 1：在 Go 代码中硬编码填入你（考官）提供的真实 GitHub Token，执行脚本。
  - [ ] 预期结果 1：打开浏览器登录考官的 GitHub，进入 Gist 页面，能看到生成了一个全新的 Gist，包含名为 `sync_manifest.json` 的空文件（或带有初始化花括号 `{}` 的文件）。

### 🟢 Quest 3.2：向 Gist 写入/读取指定文件

- **🎯 关卡目标**：封装通用 API，允许代码向刚才找到的 Gist 里面增删查改任意名称的文件内容。
- **🛠️ 推荐工具/轮子**：同上。
- **🚫 边界限制 (防超纲)**：只测试纯文本的网络传输，本关严禁结合之前的 AES 加密模块。
- **✅ 考官验收标准 (PM Checklist)**：
  - [ ] 测试动作 1：运行对应的 Go 测试脚本（尝试写入 "Hello Cloud" 到 `test.txt`）。
  - [ ] 预期结果 1：刷新 GitHub Gist 网页，看到多出一个 `test.txt` 文件，内容确为 "Hello Cloud"。

## 阶段 4：业务流组装与“胶水”桥接 (The Grand Integration)

*这是最激动人心的阶段。把前三个阶段的所有孤立积木，通过 Wails 的前后端桥接机制（Bridge）串联成完整的商业软件。*

### 🟢 Quest 4.1：桥接状态：前端保存鉴权信息

- **🎯 关卡目标**：让阶段 1 画的“设置界面”真正生效，前端输入的 Token 和密码能保存到本地持久化，供 Go 后台读取。
- **🛠️ 推荐工具/轮子**：Wails 的 `Events/Bindings` 机制，Go 的 `os` 本地文件写入。
- **🚫 边界限制 (防超纲)**：只保存配置，暂时不要触发上传/下载行为。
- **✅ 考官验收标准 (PM Checklist)**：
  - [ ] 测试动作 1：打开 GUI 界面，在设置页输入 Token "ABC" 和密码 "123"，点击保存并关闭软件。
  - [ ] 测试动作 2：重新打开软件。
  - [ ] 预期结果 1：界面能自动回显刚才填入的 Token "ABC"（密码可留空或显示星号），说明鉴权状态已持久化成功。

### 🟢 Quest 4.2：实现“上传同步”完整数据流

- **🎯 关卡目标**：在前端点击要同步的目录，Go 侧读取真实文件 -> AES加密 -> 调用 Gist API 上传 -> 更新 Manifest 文件记录。
- **🛠️ 推荐工具/轮子**：无，全是前面写好的内部函数大组装。
- **✅ 考官验收标准 (PM Checklist)**：
  - [ ] 测试动作 1：在本地桌面创建一个 `test-config.txt`，里面写上明文 "PM_Secret_Data"。
  - [ ] 测试动作 2：在 GUI 软件里添加此文件（配置相对路径如 `{{HOME}}/Desktop/test-config.txt`），点击“上传同步”。
  - [ ] 预期结果 1：去 GitHub Gist 网页查看。里面出现了这个文件，但肉眼看过去是一堆完全乱码的密文。同时 `sync_manifest.json` 里多了这条文件的路径记录。

### 🟢 Quest 4.3：实现“下载覆盖”防御与拉取流

- **🎯 关卡目标**：拉取云端记录 -> 对比本地是否存在同名文件 -> 若存在则阻断并触发前端“询问替换”弹窗 -> 用户确认后 -> 解密写入。
- **🛠️ 推荐工具/轮子**：Wails 的 `WindowExecJS` 或前端弹窗组件。
- **✅ 考官验收标准 (PM Checklist)**：
  - [ ] 测试动作 1：修改桌面上刚才的 `test-config.txt` 的内容为 "Fake Data"。
  - [ ] 测试动作 2：在 GUI 软件里点击“从云端下载同步”。
  - [ ] 预期结果 1：界面弹出一个警告框：“文件已存在，是否覆盖？”。
  - [ ] 测试动作 3：点击“取消”。
  - [ ] 预期结果 2：桌面文件内容依然是 "Fake Data"。
  - [ ] 测试动作 4：再次点击同步，并在弹窗点击“确定覆盖”。
  - [ ] 预期结果 3：打开桌面的文本文件，看到内容成功变回被解密后的 "PM_Secret_Data"。
